## C4C2D — Lifecycle Polling + Verified Replay Persistence

- เพิ่ม migration `010_authoritative_raid_replays.sql`: replay เป็น immutable row แบบ one-result/one-raid
- trusted worker ส่ง simulation version, tick/count, command-stream hash และ commands พร้อมผล
- backend ตรวจ version, command whitelist, identity length/format, tick order, count limit 25,000, final `COMPLETE` และ SHA-256 ก่อนแตะ settlement
- command stream ที่ผิด hash ถูก reject ก่อน wallet/escrow เปลี่ยน
- `GET /v1/raids/{raidId}` แสดง replay readiness/version/hash และอนุญาตเฉพาะ attacker/เจ้าของเมือง

- Unity เพิ่ม `RaidLifecycleReplayController`: poll แบบ bounded exponential backoff, อ่าน public API เท่านั้น และเริ่ม presentation หลัง replay/lifecycle identity ตรงกัน

## C4C2F — Defense History + Verified Revenge

- เพิ่ม `GET /v1/raid-history/defense` แบบ defender-only, stable cursor pagination และ index ตามผู้เล่น/เวลา/raid ID
- เพิ่ม verified replay launch ที่ส่งต่อเพียง authoritative raid UUID; Unity fetch lifecycle/replay และตรวจ identity/hash ตาม flow เดิม

## C4C2G — Replay Object Storage Adapter

- เพิ่ม migration `012_replay_object_storage.sql`: PostgreSQL เก็บ provider/key/blob SHA-256/size/encoding; replay ใหม่ไม่เก็บ command stream JSONB ซ้ำ
- เพิ่ม `IRaidReplayBlobStore` และ local immutable filesystem adapter สำหรับ dev/test โดยไม่เสียค่าบริการ
- ใช้ deterministic sharded key, atomic create, gzip, bounded read/decompression และป้องกัน path traversal
- ก่อนส่ง replay ให้ Unity ตรวจ blob size, blob SHA-256, gzip, JSON command count และ canonical command-stream hash ซ้ำ
- player API ไม่คืน storage key; มีเพียง attacker/เจ้าของเมืองที่ผ่าน participant query จึงอ่าน replay ได้
- migration เป็น dual-read: replay รุ่นเก่าที่อยู่ใน PostgreSQL ยังเปิดได้ ไม่บังคับ destructive backfill
- stage blob ก่อน financial transaction เพื่อลดเวลาถือ ledger lock; orphan ที่เกิดจาก process crash จะถูกจัดการด้วย grace-period lifecycle ใน C4D Ops
