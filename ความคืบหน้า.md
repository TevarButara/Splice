## Reconciliation — ระบบที่ต้องเปลี่ยนหรือแทนที่

### Raid Outcome เดิม

ปัจจุบัน `RaidManager` มี:

- `InProgress`
- `MonstersWin`
- `FortDefends`

และเหตุผลจบ:

- Fort destroyed
- Timer expired
- Attacker eliminated

ทิศทางใหม่ต้องรองรับอย่างน้อย:

- `InProgress`
- `FullVictory`
- `Extracted`
- `Defeat`

พร้อม end reason ที่ละเอียดและไม่ผูกผลลัพธ์กับข้อความ Monster/Fort แบบเก่า

### Reward เดิม

`RaidRewardController` ปัจจุบันให้ Meta Gold เฉพาะ Fort แตก โดยคำนวณเปอร์เซ็นต์จาก storedGold และใช้ `RaidTarget.Looted` กันรับซ้ำ

ระบบนี้เหมาะกับ greybox เดิม แต่ต้องแทนด้วย:

- Loot state: Available/Carried/Secured/Banked
- Partial extraction
- Reward preview
- Transaction ID
- Settlement ครั้งเดียว
- Stake/Escrow ในขั้นหลัง

### Static RaidContext

ใช้ต่อได้ชั่วคราวสำหรับ Prototype แต่ไม่ใช่แหล่งความจริงสำหรับ production เพราะ:

- หายเมื่อปิดเกม
- ไม่รองรับ resume/reconnect
- ไม่ใช่ transaction-safe
- ป้องกัน replay/collusion ไม่ได้

### PlayerWallet

ปัจจุบันมี Meta Gold สกุลเดียวใน PlayerPrefs ต้องแยกในอนาคตเป็น:

- Meta Gold
- Premium Diamond
- War Gem/Raid Stake

Prototype ขั้นแรกยังไม่เพิ่ม IAP หรือ Premium Diamond

## Baseline Risks และ Blockers

### 1. Scene Flow ยังไม่ครบ

`RaidTargetSelectionController` ตั้งชื่อ scene เป็น `Raid_Greybox` แต่ไม่พบ scene ชื่อนี้ใน Build Settings ปัจจุบัน

ผลกระทบ:

- Target → Raid อาจโหลดไม่สำเร็จ
- ยังไม่มีเส้นทาง Main Menu → Target → Raid → Result → Raid Again ที่พิสูจน์แล้ว

### 2. BuildZone ไม่อยู่ใน Build Settings

หากใช้เป็น scene จัดฐานจริง ต้องเพิ่มใน Build Settings หรือโหลดผ่าน flow ที่กำหนด

### 3. ไม่มี Automated Tests

กติกา reward, snapshot, Defense Capacity และ outcome มีความเสี่ยง regression เมื่อเริ่มเปลี่ยนระบบ

### 4. Scene Assembly Debt

โค้ดจำนวนมากมีแล้ว แต่การอ้างอิง Inspector และ Scene flow ยังต้องตรวจใน Unity จริง

### 5. Snapshot Spawn Assumptions

`RaidSnapshotLoader` สมมติว่า prefab มี `NetworkObject` และ component ที่ต้องใช้ หาก asset ตั้งค่าไม่ครบอาจเกิด runtime exception ควรเพิ่ม validator ก่อน content scale

### 6. Economy Coupling

storedGold, in-match Gold และ Meta Gold มีชื่อใกล้กัน ต้องตั้งชื่อและขอบเขตให้ชัดก่อนเพิ่ม War Gem/Diamond

### 7. Large Prototype Scene

`Bootstrap.unity` มีขนาดใหญ่และรวม manager/UI/gameplay หลายส่วน การเพิ่มระบบใหม่โดยแก้ scene ตรงทันทีเสี่ยง reference พัง ควรเพิ่ม code foundation ก่อน แล้วประกอบทีละ component

## Design Decisions ที่ล็อกชั่วคราวสำหรับ Prototype A

เพื่อให้พัฒนาต่อได้โดยไม่รอระบบ production:

1. `Premium Diamond` ไม่ถูกปล้นและยังไม่ทำใน Prototype A
2. `War Gem` เป็น Stake Currency แยกต่างหาก แต่ทำหลัง Core Raid/Extraction เล่นได้
3. Hero ล้มแล้วกองทัพยังเล่นต่อ แต่ไม่ถือว่า Extraction สำเร็จจนกว่าจะมีกติกา Hero/porter ที่ชัด
4. Extraction ทำได้เฉพาะ checkpoint
5. เมือง MVP มีสอง entry points
6. Squad MVP มีห้าใบ
7. Army Showcase เปิดเพียง archetype/power ไม่เปิด deck ทั้งหมด
8. เมืองมี shield/daily cap ใน Async/Economy phase ไม่ทำใน Fun Proof
9. Stake เลือกภายในช่วงที่ระบบกำหนด
10. World Hunt ใช้ Fragment + progress meter ใน phase หลัง

## ขอบเขตขั้น 1 ที่เสนอ — Raid Contract Foundation

ขั้น 1 ควรแก้เฉพาะ contract/state ที่จำเป็นสำหรับเกมใหม่ โดยยังไม่ทำ Hero, Loot object หรือ Stake จริง

### งานโค้ด

1. ปรับ `RaidOutcome` ให้รองรับ:
   - InProgress
   - FullVictory
   - Extracted
   - Defeat
2. ปรับ `RaidEndReason` ให้รองรับ:
   - CoreDestroyed
   - ExtractionCompleted
   - TimerExpired
   - AttackerEliminated
   - Hero/Objective-specific reason เผื่ออนาคตโดยไม่ทำ logic เกินจำเป็น
3. ทำ public server-only API สำหรับจบ raid ด้วย Extraction
4. รักษา Fort destroyed, timer และ elimination behavior เดิม
5. ปรับ `RaidResultUI` และ `RaidRewardController` ให้ compile และ map ผลลัพธ์ใหม่ โดย reward ยังใช้กติกาเดิมชั่วคราวเฉพาะ FullVictory
6. เพิ่ม pure result policy/testable helper เท่าที่เหมาะสม
7. เพิ่ม EditMode test foundation หาก dependency/assembly setup รองรับ

### ไม่ทำในขั้น 1

- Hero movement
- Loot objects
- Extraction Point scene object
- Three Rings
- War Gem/Stake
- World Map
- IAP

### Acceptance Criteria

- โปรเจกต์ compile
- Core แตก = FullVictory/CoreDestroyed
- Timer หมด = Defeat/TimerExpired
- Attacker eliminated = Defeat/AttackerEliminated
- มี API ที่ server เรียกเพื่อจบเป็น Extracted/ExtractionCompleted
- Event จบ raid ยิงครั้งเดียว
- FullVictory ยังให้ reward greybox เดิม
- Extraction/Defeat ไม่ให้ reward แบบ FullVictory
- UI แสดงผลใหม่ถูกต้อง

# ขั้น 1 — Raid Contract Foundation

**วันที่:** 20 กรกฎาคม 2026  
**สถานะ:** ผ่าน Play Mode acceptance ครบ 4 กรณี  
**ประเภทงาน:** Gameplay contract migration

## เป้าหมาย

เปลี่ยนผลลัพธ์ raid จากคำศัพท์ระบบเก่า `MonstersWin/FortDefends` ให้รองรับทิศทางใหม่:

- Full Victory เมื่อ Core แตก
- Successful Extraction
- Defeat จาก timer หรือกองทัพถูกกำจัด

โดยยังรักษา combat, timer, elimination และ greybox reward เดิมให้ทำงานต่อ

## Design Decisions

1. Outcome บอกผลระดับสูง ส่วน End Reason บอกสาเหตุ
2. Extraction ต้องจบจาก API ฝั่ง server เท่านั้น
3. ขั้น 1 ยังไม่ให้ reward จาก Extraction เพราะยังไม่มี secured loot
4. Full Victory ยังคงจ่าย Meta Gold แบบ greybox เดิมเพื่อไม่ให้ loop ปัจจุบันขาด

## Loot Contract

จำนวน loot ถูกแบ่งเป็นสี่ bucket:

- `Available` — ยังอยู่ในเมือง
- `Carried` — ผู้บุกกำลังแบก ถ้าล้มต้อง Drop
- `Secured` — ส่งถึง checkpoint/cache แล้ว
- `Banked` — raid จบและอนุมัติให้เข้ากระเป๋าผู้เล่นแล้ว

เส้นทางปกติ:

`Available → Carried → Secured → Banked`

เส้นทางเมื่อตัวแบกล้ม:

`Carried → Available`

Settlement:

- Full Victory: Bank `Available + Carried + Secured`
- Extracted: Bank เฉพาะ `Secured`
- Defeat: Bank 0

# Step 4C-A — Test A Hotfix: Tower Selection

วันที่: 2026-07-21

สถานะ: **Test A ผ่านแล้ว**

## อาการที่พบ

- กด `R` แล้วไม่มี feedback ที่มองเห็นได้
- คลิก Tower แล้วขึ้น `INVALID FOCUS TARGET`

## Root Cause ที่ตรวจพบผ่าน Unity MCP

- Tower ใน Bootstrap มี `TowerCharacter`, `NetworkObject` และ `MeshRenderer` ครบ
- แต่ Tower ทั้ง 10 จุดไม่มี `Collider` บน root หรือส่วนโมเดล
- ระบบเดิมใช้ `Physics.RaycastAll` จึงมองเห็น Tower บนจอแต่ pointer ray เลือกตัว Tower ไม่ได้

## สิ่งที่แก้

- เพิ่มการเลือกเป้าหมายสำรองด้วย `Renderer.bounds.IntersectRay`
- ยังคงใช้ Collider raycast เป็นลำดับแรก เพื่อไม่เปลี่ยนพฤติกรรม object ที่ตั้ง collider ถูกต้องแล้ว
- Tower/Garrison ที่ไม่มี Collider จึงเลือกได้จากขอบเขตโมเดลที่มองเห็นจริง
- ขณะอยู่ในโหมดเลือก ถ้า pointer อยู่บนพื้นหรือเป้าหมายที่เลือกไม่ได้ จะแสดงวงสีแดงที่จุด pointer
- เมื่อ pointer อยู่บน Tower/Garrison ที่เลือกได้ วงจะเปลี่ยนเป็นสีเขียว
- ไม่เพิ่ม Collider runtime จึงไม่กระทบ physics, pathfinding, projectile หรือ network prefab

## เป้าหมาย

ต่อยอด Focus Target ของ Hero ให้สามารถสั่ง “กองกำลังที่อยู่ใกล้ตัวตอนออกคำสั่ง” ช่วยโจมตีเป้าหมายเดียวกันได้ โดยไม่เปลี่ยน lane AI เป็นระบบ RTS เต็มรูปแบบ และต้องไม่มีคำสั่งค้างเมื่อเป้าหมายไปไม่ถึง

## Design Contract ที่ใช้

- Server เป็นผู้ตรวจเป้าหมายและเลือกยูนิตที่รับคำสั่ง
- รับเฉพาะ `MonsterCharacter` ฝ่าย Attacker ที่ยังมีชีวิต
- รับคำสั่งแบบ snapshot เฉพาะยูนิตที่อยู่ในรัศมี Hero ตอนยืนยันเป้าหมาย
- Garrison, ฝ่าย Defender และ Miner ไม่รับคำสั่งนี้
- คำสั่ง Squad มี priority เหนือ roadside aggro และ lane movement ชั่วคราว
- ยูนิตที่กำลังตีอยู่จะจบ attack swing ปัจจุบันก่อนเปลี่ยนพฤติกรรม เพื่อไม่ตัด animation/damage กลางคัน
- Melee ใช้ attack range เดิม, Ranged ใช้ projectile เดิม และ Supporter ยังชาร์จมานา/ใช้ spell ตามระบบเดิม
- เมื่อคำสั่งจบ ยูนิต snap กลับ waypoint ที่ใกล้ที่สุดด้านหน้าและกลับ lane AI เดิม

## Core Rule ที่ตกผลึก

- `Checkout` = ซื้อ/คืนเงินและบันทึก Draft เมืองล่าสุด
- `Deploy Snapshot` = ตรวจ Draft ที่ Checkout แล้ว จากนั้น publish เป็นเมืองตั้งรับ
- Snapshot ทุกฉบับมี immutable `snapshotId` และ revision ของตัวเอง
- การ Deploy revision ใหม่ไม่แก้ข้อมูล revision เก่า
- raid ที่จับคู่กับ snapshot ID เก่าจึงอ่านผังเดิมต่อได้ แม้เจ้าของกำลังแก้เมืองหรือ Deploy revision ใหม่
- local index แสดง history ล่าสุด 5 revision แต่ record เก่ายังคง load ด้วย snapshot ID ได้ ไม่ถูกลบตอนเลื่อน history

## Snapshot Data ที่บันทึก

- schema version, snapshot ID, revision และเวลา UTC
- owner account, faction และ base level
- deep copy ของ tower, upgrades, garrison, miner และ stored gold
- used/max defense capacity
- Base Power Rating
- validation rules version และ tactical warnings
- matchmaking eligibility
- reserve fields สำหรับ Army Showcase preset และ Hero Appearance ในขั้นต่อไป

# Prototype B — Step 6B-A: Target Pool Core

วันที่: 2026-07-22

สถานะ: **IMPLEMENTED — รอ Manual Diagnostic Test**

## เป้าหมาย

- รวมเมือง Player Snapshot และ Bot fallback ไว้ใน target pool เดียวกัน
- ให้ Player target อ้างอิง immutable `snapshotId` แทนการถือ Draft `BaseLayout`
- มีเป้าบุกได้ตลอดแม้ยังไม่มี snapshot จากผู้เล่นอื่น
- ป้องกันบุกเมืองบัญชีตัวเอง
- เตรียม metadata สำหรับ Target Card ในภายหลัง โดยยังไม่สร้าง UI ใหม่

## Target Model ใหม่

ขยาย `RaidTarget` ให้รองรับ:

- `RaidTargetSource.Bot`
- `RaidTargetSource.PlayerSnapshot`
- target ID และ snapshot ID/revision
- owner account และ faction
- Base Level / Base Power
- Used/Max Defense Capacity
- จำนวน Tower/Garrison
- Stored Gold preview
- validation version และ matchmaking eligibility
- inspection-only state สำหรับเมืองของบัญชีปัจจุบัน

กติกาสำคัญ:

- Bot target ถือ runtime `BaseLayout` ได้ เพราะเป็น fallback ที่สร้างเฉพาะ session
- Player Snapshot target ไม่เก็บ `layout` ไว้ใน target object
- เมื่อเลือกหรือเข้า Raid ต้องเรียก `TownSnapshotStore.LoadById(snapshotId)`
- หาก record ของ snapshot หายหรือไม่ eligible จะเลือกเป้าไม่ได้

## Target Pool Composition

เพิ่ม `RaidTargetPool.Compose()`:

1. รับ deployed snapshots และ bot fallbacks
2. Reject snapshot ที่ไม่มี ID/revision/owner/faction/layout/defense หรือ matchmaking disabled
3. ใส่ remote player snapshots ก่อน
4. เติม Bot จนครบ `desiredRaidableCount`
5. เมืองของบัญชีปัจจุบันถูก append หลังเป้าบุกทั้งหมดเป็น inspection-only
6. Dedupe ทุกเป้าด้วย target ID
7. คืน report จำนวน Player/Bot/Inspection/Rejected และ warnings

การเรียงแบบนี้ทำให้ปุ่มเป้าเดิมลำดับแรกยังเป็นเป้าที่บุกได้ครบ แม้ pool จะมีเมืองของตัวเองสำหรับตรวจสอบอยู่ด้วย

## RaidContext และ Loader

- เพิ่ม `RaidContext.TrySelectTarget()` เป็นจุดเลือกเป้าเดียว
- ตรวจ self-target, eligibility, looted state และ snapshot availability ก่อน lock
- เก็บ `TargetSnapshotId` และ `TargetSnapshotRevision` ข้าม scene
- `RaidSnapshotLoader` เรียก `ResolveTargetLayout()`
- Player target โหลด record ด้วย snapshot ID เท่านั้น และไม่อ่าน `PlayerBaseStore` Draft
- Bot target ยังใช้ generated layout เดิมเพื่อรักษา fallback/cold-start flow

## Provider เดิมที่อัปเกรด

`RaidTargetProvider` รุ่นเดิมถูกต่อยอดแทนการสร้าง provider ซ้ำ:

- อ่าน latest deployed snapshot ของทุก faction ใน Registry
- สร้าง Bot แบบ seeded โดยไม่เปลี่ยน Unity global Random state
- คำนวณ metadata Power/Tower/Garrison/Capacity/Gold ให้ Bot
- เติม Bot ให้มี raidable targets ครบตาม `targetCount`
- มี `LastBuildResult` สำหรับ diagnostic และ Target UI ขั้นถัดไป

## Target Button Compatibility

ปุ่ม greybox เดิมยังใช้งานได้ และแสดงข้อมูลเพิ่ม:

# Scene Architecture Refactor — Raid Arena + Role Presentations

วันที่: 2026-07-22

สถานะ: **IMPLEMENTED — automated tests และ runtime smoke ผ่าน**

## โครงสร้างใหม่

- `Bootstrap.unity` เหลือหน้าที่เป็น entry scene แล้วโหลด `RaidArena.unity`
- `RaidArena.unity` ถือ combat/network/snapshot/lane/units เพียงชุดเดียว จึงไม่มีการจำลอง raid ซ้ำสองฉาก
- แยกมุมและ UI ฝั่งผู้เล่นเป็น additive local presentation:
  - `RaidAttackerPresentation.unity`
  - `RaidDefenderPresentation.unity`
- `RaidSceneCompositionController` โหลด presentation ตามบทบาทและรับรองว่ามีกล้อง/AudioListener ใช้งานเพียงตัวเดียว
- Build Settings เรียง `Bootstrap → RaidArena → AttackerPresentation → DefenderPresentation`
- Content Validator ตรวจ scene contract, Build Settings และ Bootstrap index อัตโนมัติ

## กล้อง Defender ตาม MonCamera แบบกลับฝั่ง

- ยกเลิกการใช้ FortCamera มุมสูงเดิมและลบ FortCamera/MonCamera legacy ออกจาก RaidArena
- DefenderCamera ใช้ pitch, projection, FOV และ Orthographic Size เดียวกับ MonCamera
- กลับ yaw 180° และ mirror ตำแหน่งจาก Hero spawn ไปยัง Fort Core
- ค่าที่ตรวจได้ตอน runtime:
  - position `(7.02, 45.66, 75.19)`
  - rotation `(51.20, 178.97, 0.00)`
  - orthographic size `7`, FOV `57`
- AttackerCamera ใช้ต้นแบบ MonCamera และเปิด Hero follow ตามบทบาทฝ่ายบุก

## Regression ที่พบและแก้

- เมื่อถอดกล้อง legacy แล้ว `SideSelectionController` ยังแตะ destroyed `monsterCamera` reference ทำให้ Incoming Raid ค้างก่อน spawn
- แก้เส้นทาง Scene Composition ไม่ให้พึ่งกล้อง legacy และใช้ Unity-safe null check แทน null-conditional
- เพิ่ม test `IncomingDefenseView_IgnoresDestroyedLegacyCameraReference`
- เพิ่ม test mirror camera และ scene architecture/build settings contract
