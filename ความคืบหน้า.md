## Reconciliation — ระบบที่ต้องเปลี่ยนหรือแทนที่

### Raid Outcome เดิม

ปัจจุบัน `RaidManager` มี:

- `InProgress`
- `MonstersWin`
- `FortDefends`

และเหตุผลจบ:

- Fort destroyed
- Timer expired
- Attacker eliminated

ทิศทางใหม่ต้องรองรับอย่างน้อย:

- `InProgress`
- `FullVictory`
- `Extracted`
- `Defeat`

พร้อม end reason ที่ละเอียดและไม่ผูกผลลัพธ์กับข้อความ Monster/Fort แบบเก่า

### Reward เดิม

`RaidRewardController` ปัจจุบันให้ Meta Gold เฉพาะ Fort แตก โดยคำนวณเปอร์เซ็นต์จาก storedGold และใช้ `RaidTarget.Looted` กันรับซ้ำ

ระบบนี้เหมาะกับ greybox เดิม แต่ต้องแทนด้วย:

- Loot state: Available/Carried/Secured/Banked
- Partial extraction
- Reward preview
- Transaction ID
- Settlement ครั้งเดียว
- Stake/Escrow ในขั้นหลัง

### Static RaidContext

ใช้ต่อได้ชั่วคราวสำหรับ Prototype แต่ไม่ใช่แหล่งความจริงสำหรับ production เพราะ:

- หายเมื่อปิดเกม
- ไม่รองรับ resume/reconnect
- ไม่ใช่ transaction-safe
- ป้องกัน replay/collusion ไม่ได้

### PlayerWallet

ปัจจุบันมี Meta Gold สกุลเดียวใน PlayerPrefs ต้องแยกในอนาคตเป็น:

- Meta Gold
- Premium Diamond
- War Gem/Raid Stake

Prototype ขั้นแรกยังไม่เพิ่ม IAP หรือ Premium Diamond

## Baseline Risks และ Blockers

### 1. Scene Flow ยังไม่ครบ

`RaidTargetSelectionController` ตั้งชื่อ scene เป็น `Raid_Greybox` แต่ไม่พบ scene ชื่อนี้ใน Build Settings ปัจจุบัน

ผลกระทบ:

- Target → Raid อาจโหลดไม่สำเร็จ
- ยังไม่มีเส้นทาง Main Menu → Target → Raid → Result → Raid Again ที่พิสูจน์แล้ว

### 2. BuildZone ไม่อยู่ใน Build Settings

หากใช้เป็น scene จัดฐานจริง ต้องเพิ่มใน Build Settings หรือโหลดผ่าน flow ที่กำหนด

### 3. ไม่มี Automated Tests

กติกา reward, snapshot, Defense Capacity และ outcome มีความเสี่ยง regression เมื่อเริ่มเปลี่ยนระบบ

### 4. Scene Assembly Debt

โค้ดจำนวนมากมีแล้ว แต่การอ้างอิง Inspector และ Scene flow ยังต้องตรวจใน Unity จริง

### 5. Snapshot Spawn Assumptions

`RaidSnapshotLoader` สมมติว่า prefab มี `NetworkObject` และ component ที่ต้องใช้ หาก asset ตั้งค่าไม่ครบอาจเกิด runtime exception ควรเพิ่ม validator ก่อน content scale

### 6. Economy Coupling

storedGold, in-match Gold และ Meta Gold มีชื่อใกล้กัน ต้องตั้งชื่อและขอบเขตให้ชัดก่อนเพิ่ม War Gem/Diamond

### 7. Large Prototype Scene

`Bootstrap.unity` มีขนาดใหญ่และรวม manager/UI/gameplay หลายส่วน การเพิ่มระบบใหม่โดยแก้ scene ตรงทันทีเสี่ยง reference พัง ควรเพิ่ม code foundation ก่อน แล้วประกอบทีละ component

## Design Decisions ที่ล็อกชั่วคราวสำหรับ Prototype A

เพื่อให้พัฒนาต่อได้โดยไม่รอระบบ production:

1. `Premium Diamond` ไม่ถูกปล้นและยังไม่ทำใน Prototype A
2. `War Gem` เป็น Stake Currency แยกต่างหาก แต่ทำหลัง Core Raid/Extraction เล่นได้
3. Hero ล้มแล้วกองทัพยังเล่นต่อ แต่ไม่ถือว่า Extraction สำเร็จจนกว่าจะมีกติกา Hero/porter ที่ชัด
4. Extraction ทำได้เฉพาะ checkpoint
5. เมือง MVP มีสอง entry points
6. Squad MVP มีห้าใบ
7. Army Showcase เปิดเพียง archetype/power ไม่เปิด deck ทั้งหมด
8. เมืองมี shield/daily cap ใน Async/Economy phase ไม่ทำใน Fun Proof
9. Stake เลือกภายในช่วงที่ระบบกำหนด
10. World Hunt ใช้ Fragment + progress meter ใน phase หลัง

## ขอบเขตขั้น 1 ที่เสนอ — Raid Contract Foundation

ขั้น 1 ควรแก้เฉพาะ contract/state ที่จำเป็นสำหรับเกมใหม่ โดยยังไม่ทำ Hero, Loot object หรือ Stake จริง

### งานโค้ด

1. ปรับ `RaidOutcome` ให้รองรับ:
   - InProgress
   - FullVictory
   - Extracted
   - Defeat
2. ปรับ `RaidEndReason` ให้รองรับ:
   - CoreDestroyed
   - ExtractionCompleted
   - TimerExpired
   - AttackerEliminated
   - Hero/Objective-specific reason เผื่ออนาคตโดยไม่ทำ logic เกินจำเป็น
3. ทำ public server-only API สำหรับจบ raid ด้วย Extraction
4. รักษา Fort destroyed, timer และ elimination behavior เดิม
5. ปรับ `RaidResultUI` และ `RaidRewardController` ให้ compile และ map ผลลัพธ์ใหม่ โดย reward ยังใช้กติกาเดิมชั่วคราวเฉพาะ FullVictory
6. เพิ่ม pure result policy/testable helper เท่าที่เหมาะสม
7. เพิ่ม EditMode test foundation หาก dependency/assembly setup รองรับ

### ไม่ทำในขั้น 1

- Hero movement
- Loot objects
- Extraction Point scene object
- Three Rings
- War Gem/Stake
- World Map
- IAP

### Acceptance Criteria

- โปรเจกต์ compile
- Core แตก = FullVictory/CoreDestroyed
- Timer หมด = Defeat/TimerExpired
- Attacker eliminated = Defeat/AttackerEliminated
- มี API ที่ server เรียกเพื่อจบเป็น Extracted/ExtractionCompleted
- Event จบ raid ยิงครั้งเดียว
- FullVictory ยังให้ reward greybox เดิม
- Extraction/Defeat ไม่ให้ reward แบบ FullVictory
- UI แสดงผลใหม่ถูกต้อง

# ขั้น 1 — Raid Contract Foundation

**วันที่:** 20 กรกฎาคม 2026  
**สถานะ:** ผ่าน Play Mode acceptance ครบ 4 กรณี  
**ประเภทงาน:** Gameplay contract migration

## เป้าหมาย

เปลี่ยนผลลัพธ์ raid จากคำศัพท์ระบบเก่า `MonstersWin/FortDefends` ให้รองรับทิศทางใหม่:

- Full Victory เมื่อ Core แตก
- Successful Extraction
- Defeat จาก timer หรือกองทัพถูกกำจัด

โดยยังรักษา combat, timer, elimination และ greybox reward เดิมให้ทำงานต่อ

## Design Decisions

1. Outcome บอกผลระดับสูง ส่วน End Reason บอกสาเหตุ
2. Extraction ต้องจบจาก API ฝั่ง server เท่านั้น
3. ขั้น 1 ยังไม่ให้ reward จาก Extraction เพราะยังไม่มี secured loot
4. Full Victory ยังคงจ่าย Meta Gold แบบ greybox เดิมเพื่อไม่ให้ loop ปัจจุบันขาด

## Loot Contract

จำนวน loot ถูกแบ่งเป็นสี่ bucket:

- `Available` — ยังอยู่ในเมือง
- `Carried` — ผู้บุกกำลังแบก ถ้าล้มต้อง Drop
- `Secured` — ส่งถึง checkpoint/cache แล้ว
- `Banked` — raid จบและอนุมัติให้เข้ากระเป๋าผู้เล่นแล้ว

เส้นทางปกติ:

`Available → Carried → Secured → Banked`

เส้นทางเมื่อตัวแบกล้ม:

`Carried → Available`

Settlement:

- Full Victory: Bank `Available + Carried + Secured`
- Extracted: Bank เฉพาะ `Secured`
- Defeat: Bank 0
